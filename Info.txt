================================================================================
                    UniScraper — Comprehensive Project Documentation
================================================================================

Welcome to the UniScraper project! This document provides a complete overview of the project's architecture, directory structure, data flow, external integrations, and setup instructions.

================================================================================
                                1. TECH STACK
================================================================================

- Frontend:  React 19, TanStack Router/Query/Start, Tailwind CSS v4, shadcn/ui
- Backend:   FastAPI, Python 3.11+, asyncio
- Database:  MongoDB Atlas (Motor async driver)
- Crawling:  Tier 1: httpx + Playwright, Tier 2: Firecrawl, Tier 3: Crawl4AI, Jina Reader
- Discovery: BFS crawler + SerpAPI + Jina Search
- LLMs:      Gemini 2.5 Flash → Groq → Ollama (fallback chain)
- Parsing:   BeautifulSoup + lxml

================================================================================
                          2. ARCHITECTURE & PIPELINE FLOW
================================================================================

The application is divided into a React frontend and a FastAPI backend.

[ Frontend (React 19 + TanStack Router) ]
│
├── /            : Single URL scraper (Phase 1)
├── /discover    : University search + program discovery (Phase 2)
├── /batch       : Multi-URL batch scraper
└── /history     : Past results archive
│
[ FastAPI Backend (Python 3.11+) ]
│
├── Phase 2: Discovery Pipeline
│     ├── Domain Resolver        : Heuristic + SerpAPI fallback
│     ├── Program Discovery      : BFS crawler with scoring
│     ├── SerpAPI Client         : Fallback for WAF-blocked sites
│     ├── Jina Search            : Fast, AI-driven program discovery and search
│     └── Discovery Orchestrator : Async execution + 24h MongoDB cache
│
└── Phase 1: Scrape Pipeline
      ├── Three-Tier Intelligent Fetching
      │     ├── Tier 1: Custom (httpx → Playwright)   [FAST]
      │     ├── Tier 2: Firecrawl                     [CLOUDFLARE]
      │     ├── Tier 3: Crawl4AI                      [DEEP CRAWL]
      │     └── Tier 4: Jina Reader                   [CLEAN MARKDOWN]
      │
      ├── Page Classifier     : Identifies page types (admissions / english / tuition / etc.)
      ├── AI Extractor (P1)   : Uses Gemini 2.5 Flash → Groq → Ollama
      ├── Gap Analyzer        : Detects missing critical fields
      ├── Targeted Recrawl    : Fetches 2–5 more pages if gaps are found
      ├── AI Extractor (P2)   : Re-extraction with new pages
      ├── Regex Extractor     : Pre/post-LLM fallback layer
      ├── Field Validator     : Semantic sanity checks
      └── MongoDB             : Saves results + 24h cache

                    4. API ENDPOINTS & DATA FLOW
================================================================================

This section explains how the frontend and backend communicate for different operations.

[ Client / React UI ]
│
├── 📌 PHASE 1: Single URL Scraping (Extracting details of a single program)
│   │
│   ├── POST /api/v1/scrape
│   │    └── Flow: Client sends URL -> DB creates record -> Background Task starts -> Returns 'scrape_id'
│   │
│   ├── GET /api/v1/scrape/{id}
│   │    └── Flow: Client polls every 3s -> Backend checks status -> Returns extracted JSON on success
│   │
│   └── DELETE /api/v1/scrape/{id}
│        └── Flow: Client requests deletion -> DB removes scrape record
│
├── 📌 PHASE 2: University Discovery (Finding all academic programs)
│   │
│   ├── POST /api/v1/discover
│   │    └── Flow: Client sends Uni name -> Domain resolved -> Crawler starts -> Returns 'discovery_id'
│   │
│   └── GET /api/v1/discover/{id}
│        └── Flow: Client polls status -> Backend returns list of discovered program URLs
│
├── 📌 BATCH PROCESSING (Scraping multiple URLs simultaneously)
│   │
│   ├── POST /api/v1/scrapes/batch
│   │    └── Flow: Client sends URL array -> Tasks queued with stagger -> Returns 'batch_id'
│   │
│   └── GET /api/v1/batch/{id}
│        └── Flow: Client polls batch progress -> Backend returns metrics (done, pending, failed)
│
└── 📌 DATA EXPORT & HISTORY
    │
    ├── GET /api/v1/scrapes               (Retrieves a paginated history of past scrapes)
    ├── GET /api/v1/scrapes/export/csv    (Bulk downloads the scraped data in CSV format)
    └── GET /api/v1/scrapes/export/json   (Downloads the scraped data in raw JSON format)

================================================================================
                 5. EXTERNAL APIs & AI SERVICES (Third-Party Tools)
================================================================================

Alongside the core FastAPI backend, this project leverages several third-party APIs and AI services for advanced scraping and data extraction.

🧠 1. Gemini (Google AI)
   • Role: Primary Large Language Model (LLM).
   • Use Case: It processes messy, unstructured HTML text to intelligently extract fees, deadlines, IELTS requirements, and formats the output into structured JSON.

⚡ 2. Groq
   • Role: High-speed AI API (LLM).
   • Use Case: Acts as the first fallback. If the Gemini API hits a rate limit or encounters an error, the system automatically routes the extraction task to Groq.

🖥️ 3. Ollama
   • Role: Local AI model runner.
   • Use Case: Acts as the ultimate fallback. If both Gemini and Groq fail, the system utilizes a smaller, locally hosted AI model to complete the extraction.

🕷️ 4. Firecrawl
   • Role: Advanced web scraping API.
   • Use Case: Operates as the "Tier 2 Fetcher". When a university website blocks standard scraping tools (like Playwright) via Cloudflare or other WAF protections, Firecrawl is used to bypass these firewalls and retrieve the data.

🤖 5. Crawl4AI (Currently inactive per team notes)
   • Role: Open-source deep crawler API.
   • Use Case: Operates as the "Tier 3 Fetcher". It is designed to render heavy JavaScript pages and extract clean text.

🔍 6. SerpAPI
   • Role: Google Search API.
   • Use Case: Utilized in Phase 2 (Discovery). If our internal crawler cannot navigate a university's site effectively, SerpAPI is used to search Google directly for the university's academic programs.

🔗 7. Jina AI (Search & Reader)
   • Role: AI-driven search and content extraction APIs.
   • Use Case: Utilized in Phase 2 (Discovery) to rapidly find relevant program pages using `s.jina.ai`, and fetching clean markdown content using `r.jina.ai`.

                          6. ENVIRONMENT VARIABLES (.env)
================================================================================

File location: uniscraper-backend\.env

REQUIRED VARIABLES:
  GEMINI_API_KEY=your_gemini_key          # Primary LLM (Gemini 2.5 Flash)
  MONGODB_URI=your_mongodb_atlas_uri      # MongoDB Atlas connection string
  SERPAPI_KEY=your_serpapi_key            # Used for program discovery (Free tier compatible)

OPTIONAL VARIABLES:
  GROQ_API_KEY=your_groq_key              # Fallback LLM
  FIRECRAWL_API_KEY=your_firecrawl_key    # Tier 2 fetching API
  CRAWL4AI_ENABLED=true                   # Enables Tier 3 fetching
  JINA_API_KEY=your_jina_key              # Jina Search and Reader API Key
  JINA_ENABLED=true                       # Enables Jina API integrations
  DB_NAME=autonova_scraper                # Database name configuration
  LLM_MODEL=gemini-2.5-flash              # LLM model selection
  CORS_ORIGINS=http://localhost:5173      # Allowed CORS origins
